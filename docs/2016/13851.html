I think it's more because Lisp IDEs, ironically, are ultimately TEXT editors and not SEXP editors (including Paredit and similar).. IntelliJ does a lot of parsing/compilation to infer things about your source code. In Lisp it should be easier to work with S-expressions directly and keep additional metadata, like this expression was loaded from file x at line y with this textual representation, but Emacs/SLIME work mostly on text, not code. Which is a shame.

Re: Sadness in my heart...

Well, I was thinking about this and the problem is simply the flexibility of macros. Certainly the LW IDE and SLIME do perform some parsing of s-expressions as they're entered. But if you wanted to provide slot hints, you'd have to be able to expand any macro that reduced to defclass. And a macro can contain arbitrary code, so you need an entire LISP interpreter for that. The LW IDE should be ideally placed to do that since it is written in LW and runs LW in parallel (the same reason why IntelliJ is so good for Java, although it is getting better on other languages). But then you hit a security problem - a mischievous library could drop in a macro which performs "rm -rf ~" at expand time, and as soon as you started typing the name of that macro, the IDE would expand it for analysis and wipe out your home directory.

Mark

As far as I am concerned, it is not the language – I have written programs in many languages for different purposes. When the objectives of the program are met, the language used is mostly inconsequential as long as I didn’t have to stand on my head to make the writing productive.

 

So, to me, it is the development tool that matters more – a good one is one that does not get in my way and allows me to write tight, functional and easily testable code.

 

Hence, I still believe that the finest, easiest, and by far the most productive development environment I ever programmed in was a Symbolics 3600 in the 1980’s.

 

Still has not been equaled, IMHO.

On Sun, May 1, 2016 at 12:04 AM, Paul Tarvydas <paultarvydas@gmail.com> wrote:

I mostly agree with Pascal’s reservations about IDE’s.

AFAICT, the IDE was invented (enabled) in 1956 with the *language* Lisp 1.5..  The *language* contains the equivalents of LEX and YACC.  The *language* can act on itself.

In my first Dr. Dobb’s article (1978/9?) I published source code for a SEXP editor and debugger in lisp.

Other *languages* got IDE-envy and have been struggling to catch up.  Rather unsuccessfully, afaict.  If the *language* doesn’t have “LEX and YACC” built into it, you have to jump through hoops to get an IDE that is 50% as good as a lisp environment.

Then, there’s the Unix (shell) environment, which makes things incredibly easy and simple.   Around the same time as the Unix shell was invented, there was another development, called FBP (flow-based-programming) which is even better at piping than the Unix shell is.  (Still actively discussed and developed - see google group flow-based-programming).

The modern IDE is necessitated by inferior languages designed by syntax fascists.

We should strive to getting rid of syntax and going back to simplicity.

[Other examples of such wonderful simplicity are Rebol and the Smalltalk Blue Book].
