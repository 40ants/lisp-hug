Re: MMapper

Maybe this goes without saying, but a prime reason for our invention of the MMapper library follows from measurements which show that direct memory mapped file I/O runs about 100x faster than conventional stream based file I/O. Modern OS's implement paged memory very efficiently, obviating our implementation of a parallel layer in Lisp.

The other reason is that, by means of the implemented persistent object class, we can directly access portions of a file as though they were file resident C structs. Byte endian issues are handled automatically in MMapper to allow files written on one host to be read / written compatibly on an entirely different host. (e.g., PPC vs Intel)

It would be interesting (though off-topic, probably) to know where the stream overhead comes from.  My (somewhat informed, but for a small selection of platforms) is that you don't need to memory-map files to get very good I/O performance, for programs which do their I/O carefully.
